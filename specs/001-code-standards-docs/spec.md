# Feature Specification: Приведение кода к стандартам и документирование фич

**Feature Branch**: `[001-code-standards-docs]`
**Created**: 2026-07-20
**Status**: Draft
**Input**: User description: "Надо привести код в соответствие со стандартами, описать уже имеющиеся фичи"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Разработчик понимает стандарты кода (Priority: P1)

Разработчик (текущий или будущий контрибьютор), впервые открывая репозиторий, может
за 5 минут найти и прочитать актуальные правила оформления кода: какой стиль
именования, какие паттерны обязательны (например, `redirectErrorStream(true)` для
`ProcessBuilder`), каких анти-паттернов избегать (см. зафиксированные
инварианты в `DEVELOPMENT.md`). Правила подкреплены автоматическими линтерами,
которые блокируют merge при нарушениях.

**Why this priority**: без явного канона стандартов любой «приведение к стандартам»
вырождается в субъективный вкус автора правок. Без автоматизации правила
превращаются в пожелания. P1 — фундамент, на котором строятся остальные stories.

**Independent Test**: можно проверить, открыв `CONTRIBUTING.md` (или
`docs/code-style.md`) и запустив линтеры (`./gradlew ktlintCheck detekt` и
`cd webvue3 && npm run lint`) в чистом клоне — линтеры находят нарушения
и валит CI до тех пор, пока baseline не зафиксирован.

**Acceptance Scenarios**:
1. **Given** новый разработчик в репозитории, **When** открывает `CONTRIBUTING.md`,
   **Then** находит явные правила для Kotlin/Spring Boot и Vue 3 с примерами
   «правильно / неправильно» и ссылками на инварианты из `DEVELOPMENT.md`.
2. **Given** правила зафиксированы, **When** запускается линтер (`kt候候候
   `ktlint`/`detekt`/`eslint`/`prettier`), **Then** он валится на заранее
   размеченных примерах нарушений и проходит на «чистом» коде.
3. **Given** baseline нарушений зафиксирован, **When** разработчик делает PR,
   **Then** CI блокирует merge при появлении новых нарушений сверх baseline.

### User Story 2 - Существующий код проходит автоматическую проверку (Priority: P1)

Pre-commit-хук и CI запускают линтеры (Kotlin: `ktlint`+`detekt`;
Vue/TypeScript: `eslint`+`prettier`); текущий код доведён до состояния, когда
запуск линтера завершается с кодом 0 (или с заранее описанным baseline-списком
известных нарушений, который уменьшается со временем — целевой темп ≥10% в
месяц).

**Why this priority**: код без работающего линтера — «код с намерением», не код
со стандартом. Только P1, потому что без автоматизации правила превращаются в
пожелания.

**Independent Test**: запуск `./gradlew ktlintCheck detekt` и `cd webvue3 && npm run lint`
в чистом клоне завершается с кодом 0 (или с известным baseline, явно
задокументированным).

**Acceptance Scenarios**:
1. **Given** baseline зафиксирован, **When** разработчик делает PR, **Then** CI
   блокирует merge при появлении новых нарушений сверх baseline.
2. **Given** в код внесено изменение, нарушающее правило, **When** запускается
   линтер локально, **Then** он завершается с ненулевым кодом и понятным
   сообщением об ошибке.

### User Story 3 - Каждая ключевая фича имеет per-feature документ (Priority: P2)

Каждая ключевая фича (MLT-генератор, async-очередь `KaraokeProcess*`, двух-БД sync,
рендер MP4, SSE-уведомления, премиум-стемы, LLM-поиск текстов, Telegram-автопубликация,
мониторинг) имеет человекочитаемый per-feature документ в `docs/features/`
со стандартными разделами: «Что делает», «Зачем», «Как работает (кратко)»,
«Инварианты / правила», «Известные ловушки», «Ссылки на ключевые классы/файлы».
Документ ссылается на ключевые классы/файлы в формате `файл:строка` или
`класс#метод`.

**Why this priority**: P2, потому что DEVELOPMENT.md уже содержит «Architecture
notes» с большей частью нужной информации; задача — выделить это в per-feature
документы и обновить ссылки. Без P1 это развалится (правила оформления кода
определяют, как именно описывать фичи).

**Independent Test**: открыть `docs/features/` и убедиться, что каждая фича из
списка выше (9 фич) имеет файл со всеми требуемыми разделами и непустыми
ссылками.

**Acceptance Scenarios**:
1. **Given** фича реализована, **When** разработчик ищет её описание, **Then**
   находит файл в `docs/features/<feature-name>.md` за ≤2 минуты.
2. **Given** описание существует, **When** оно ссылается на ключевые классы/
   модули, **Then** все ссылки ведут на актуальный код (битые ссылки
   обнаруживаются линтером документации, если подключён, или вручную
   при code review).
3. **Given** per-feature документ создан, **When** разработчик читает раздел
   «Инварианты», **Then** находит все правила из `DEVELOPMENT.md`,
   относящиеся к этой фиче, без необходимости открывать `DEVELOPMENT.md`.

### User Story 4 - KDoc/JSDoc на публичных API (Priority: P3)

Публичные классы и методы (Kotlin: `public` без `internal`/`private`;
TypeScript/Vue: экспортируемые символы) сопровождаются кратким KDoc/JSDoc-
комментарием: что делает, какие ограничения, какой контракт по исключениям.
KDoc/JSDoc ссылается на per-feature документ через `@see docs/features/<slug>.md`
для архитектурного контекста.

**Why this priority**: P3, потому что без P1 (правила оформления) и P2
(per-feature документация) P4 не имеет смысла — будет разнобой стилей и
неполнота.

**Independent Test**: запуск Dokka / typedoc генерирует документацию без
«unresolved @see» / «missing description» warning'ов.

**Acceptance Scenarios**:
1. **Given** публичный класс без KDoc, **When** запускается Dokka, **Then**
   отчёт содержит предупреждение с указанием файла.
2. **Given** Dokka-сборка проходит без warning'ов, **When** разработчик
   читает сгенерированный HTML, **Then** все публичные API задокументированы
   и содержат `@see`-ссылку на соответствующий per-feature документ.

---

### Edge Cases

- Что если линтер выдаёт тысячи нарушений на старте — вводить baseline или
  исправлять всё разом? (Решение: **baseline** с темпом сокращения ≥10%/мес.)
- Что если какое-то «нарушение» на самом деле намеренное (например,
  `JSON`-ключ без `is`-префикса в Kotlin — задокументированный инвариант
  Jackson)? Решение: **baseline с inline-комментарием** `// ktlint-disable
  ... // intentional: Jackson-инвариант, см. DEVELOPMENT.md «Kotlin is*-поля»`.
- Что если описание фичи устарело (код изменился, доку нет)? Решение:
  **линтер документации** (или ручная проверка при code review) ловит
  битые ссылки; в правиле оформления — «PR, меняющий код из per-feature
  документа, обязан обновить и сам документ».
- Что если в активной разработке находится новая фича, ещё не описанная?
  Решение: **per-feature документ создаётся в том же PR, что и код фичи**
  (правило для code review).
- Что если KDoc-ссылка на per-feature документ ведёт на несуществующий
  файл? Решение: Dokka/typedoc-генерация валится на warning; правило
  оформления запрещает такие ссылки.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять единый документ с правилами оформления
  кода для Kotlin/Spring Boot и Vue 3 (`CONTRIBUTING.md` или `docs/code-style.md`),
  с явными примерами «правильно / неправильно» и ссылками на инварианты
  из `DEVELOPMENT.md` / `AGENTS.md`.
- **FR-002**: Система MUST запускать линтеры (Kotlin: `ktlint`+`detekt`;
  TypeScript/Vue: `eslint`+`prettier`) в pre-commit-хуке и в CI; нарушения
  блокируют merge.
- **FR-003**: Система MUST иметь baseline-список известных нарушений
  (`config/ktlint/baseline.xml`, `config/detekt/baseline.xml`,
  `.eslintrc-baseline.json`); baseline сокращается на ≥10% в месяц;
  новые нарушения сверх baseline блокируют merge.
- **FR-004**: Система MUST иметь per-feature документацию в `docs/features/`
  для всех 9 ключевых подсистем активного кода: MLT-генератор,
  async-очередь `KaraokeProcess*`, двух-БД sync, рендер MP4, SSE-уведомления,
  премиум-стемы, LLM-поиск текстов, Telegram-автопубликация, мониторинг.
- **FR-005**: Каждый per-feature документ MUST содержать разделы: «Что делает»,
  «Зачем», «Как работает (кратко)», «Инварианты / правила», «Известные ловушки»,
  «Ссылки на ключевые классы/файлы».
- **FR-006**: Публичные API MUST сопровождаться KDoc/JSDoc-комментариями,
  генерирующими документацию через Dokka / typedoc без warning'ов;
  KDoc/JSDoc MUST содержать `@see`-ссылку на соответствующий
  per-feature документ.
- **FR-007**: Существующие правила из `AGENTS.md` и инварианты из `DEVELOPMENT.md`
  MUST быть перенесены в `CONTRIBUTING.md` и в per-feature документы
  (без потери информации) с явными ссылками на источник.
- **FR-008**: Документация MUST переживать рефакторинг кода: ссылки
  `файл:строка` / `класс#метод` обязаны валидироваться (линтером или
  вручную при code review); битые ссылки исправляются в том же PR.
- **FR-009**: PR, вводящий новую ключевую фичу или меняющий поведение
  существующей из списка FR-004, MUST в том же PR обновлять (или создавать)
  соответствующий per-feature документ. Правило проверяется на code review.
- **FR-010**: Изменения в этом скоупе MUST проходить review на соответствие
  `constitution.md` (NON-NEGOTIABLE принципы: сырой JDBC, self-contained
  пайплайн, async-очередь, разделение admin/public).
- **FR-011**: Скоуп стандартов и документации MUST ограничиваться активным
  кодом: `karaoke-app`, `karaoke-web`, `webvue3`, `karaoke-public`, `deploy/`.
  Legacy-модули `karaoke-db` (только `Main.java`) и `karaoke-vue` (только
  `src/assets`, не в сборке) в скоуп не входят и могут быть удалены
  в рамках отдельной задачи.

### Key Entities

- **Стандарт оформления кода** (`CONTRIBUTING.md` или `docs/code-style.md`):
  атрибуты — язык, раздел (Kotlin/Vue/JSON/SQL), правило, пример «правильно»,
  пример «неправильно», ссылка на инвариант в `DEVELOPMENT.md`/`AGENTS.md`.
- **Per-feature документ** (`docs/features/<slug>.md`): атрибуты — название
  фичи, slug, статус (active/deprecated/experimental), 6 обязательных разделов,
  ссылки на ключевые классы в формате `файл:строка`.
- **Baseline нарушений** (`config/ktlint/baseline.xml`, `config/detekt/baseline.xml`,
  `.eslintrc-baseline.json`): атрибуты — путь к файлу, количество нарушений,
  дата последнего сокращения, целевой темп.
- **Конституция** (`.specify/memory/constitution.md`): действующий набор
  непреложных принципов; MUST обновляться по governance при изменении
  стандартов.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% ключевых фич (9 подсистем из FR-004) имеют per-feature
  документ в `docs/features/` со всеми 6 обязательными разделами к моменту
  закрытия фичи.
- **SC-002**: Pre-commit-хук + CI запускают линтеры и блокируют merge при
  новых нарушениях сверх baseline; baseline сокращается на ≥10% в месяц
  (замеряется diff'ом baseline-файлов между релизами).
- **SC-003**: Время онбординга нового разработчика (от клона до первого
  осмысленного PR) сокращается на 30% по сравнению с текущим
  (замеряется опросом / логом активности).
- **SC-004**: ≥90% публичных API имеют KDoc/JSDoc; Dokka/typedoc генерируют
  документацию без warning'ов.
- **SC-005**: 0 битых ссылок «файл:строка» в per-feature документах
  (проверяется линтером документации или вручную при code review каждого
  PR, трогающего доку).
- **SC-006**: Каждый существующий инвариант из `DEVELOPMENT.md` имеет явную
  ссылку в `CONTRIBUTING.md` или в per-feature документе; в самом
  `DEVELOPMENT.md` сохраняется краткое summary, но полный текст перенесён
  в `docs/`.
- **SC-007**: 0 PR с новой/изменённой ключевой фичой из FR-004 принимается
  без обновления/создания соответствующего per-feature документа
  (правило FR-009 соблюдается на 100% code review'ом).

## Assumptions

- Стандарты оформления уже частично зафиксированы в `AGENTS.md` (русский
  язык, стиль Kotlin/Vue, безопасные паттерны) и в `DEVELOPMENT.md`
  (инварианты, ловушки, dated-история). Задача — канонизировать и
  автоматизировать проверку, а не выдумывать с нуля.
- Команда / сообщество готово принять baseline-подход (фиксировать известные
  нарушения и уменьшать их постепенно), а не требовать «исправить всё
  разом».
- Внедрение линтеров не должно ломать текущий пайплайн сборки: добавление
  в CI — отдельной стадией, не в горячий путь `bootJar`.
- Документирование фич и `CONTRIBUTING.md` ведутся на русском (согласно
  `AGENTS.md`).
- Скоуп ограничен активным кодом: `karaoke-app`, `karaoke-web`, `webvue3`,
  `karaoke-public`, `deploy/`. Legacy `karaoke-db` и `karaoke-vue` — вне
  скоупа (могут быть удалены в отдельной задаче).
- «Ключевые фичи» — это 9 подсистем, перечисленных во FR-004. Список
  фиксируется на этапе планирования; расширение — через `clarify`-итерацию.
- Под «активными линтерами» подразумеваются: Kotlin — `ktlint` (форматирование)
  + `detekt` (code smells); Vue/TS — `eslint` (правила) + `prettier`
  (форматирование). Альтернативные инструменты допустимы, если они
  решают те же задачи и совместимы с baseline-подходом.

## Clarifications

### Session 2026-07-20

- Q: Что считать «стандартами»? → A: **B — AGENTS.md + автоматические линтеры
  (ktlint, detekt, eslint, prettier) с baseline-подходом.** Документ фиксирует
  правила, инструменты их проверяют в pre-commit + CI, baseline сокращается
  постепенно (≥10%/мес).
- Q: Какие модули входят в скоуп? → A: **B — весь активный код:**
  `karaoke-app` + `karaoke-web` + `webvue3` + `karaoke-public` + `deploy/`.
  Legacy `karaoke-db` и `karaoke-vue` — вне скоупа.
- Q: В каком формате описывать фичи? → A: **B — per-feature документы в
  `docs/features/<slug>.md`** (6 обязательных разделов: «Что / Зачем / Как /
  Инварианты / Ловушки / Ссылки») + ссылки на них из KDoc/JSDoc через `@see`
  для архитектурного контекста.
