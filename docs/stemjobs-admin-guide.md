# «Создать минусовку из аудиофайла» — руководство администратора

Премиум-фича личного кабинета (karaoke-public): пользователь загружает произвольный аудиофайл и
получает исходник + стемы (demucs2 — музыка/голос; demucs5 — музыка/голос/бас/ударные/остальное).
Этот документ — что нужно один раз настроить перед первым запуском и как этим потом пользоваться.
Архитектурные детали и код — см. комментарии в `StemJob.kt`, `StemJobPollScheduler.kt`,
`StemJobProcessing.kt`, `StemJobCleanup.kt` (`karaoke-app`) и `PublicStemJobController.kt`,
`InternalStemJobController.kt` (`karaoke-web`).

## Как это работает (коротко)

- Пользователь загружает файл через личный кабинет → **karaoke-web** (прод) сохраняет его временно
  на своём диске и создаёт запись задания в БД.
- **karaoke-app** (эта машина, машина администратора — *единственная*, которая умеет писать в MinIO
  и запускать demucs/docker) раз в 30-60 секунд опрашивает прод-БД, забирает файл с karaoke-web по
  внутреннему HTTP-запросу и ставит задание в очередь (свой отдельный "лейн", не мешает обычному
  пайплайну выпуска песен).
- Готовые стемы заливаются в MinIO, пользователь скачивает их из личного кабинета; всё удаляется
  автоматически через 24 часа после готовности (или сразу — если пользователь или администратор
  нажал «Удалить»).

**Важно понимать: вся обработка происходит только пока запущен karaoke-app.** Это desktop-приложение
администратора, а не постоянно работающий серверный сервис (как и весь остальной пайплайн выпуска
песен). Если karaoke-app выключен — новые задания пользователей просто копятся в статусе «В очереди»
на проде и начнут обрабатываться, как только вы его запустите. То же самое касается автоматической
уборки протухших/удалённых заданий (см. ниже) — она тоже работает только пока запущен karaoke-app.

## 1. Единоразовая настройка перед первым запуском

### 1.1. Применить миграцию БД

Файл `deploy/karaoke-db/22_stem_jobs.sql` нужно применить **вручную на каждой БД отдельно** — и на
LOCAL (для локальной отладки), и на PROD (79.174.95.69:8832) — таблица `tbl_stem_jobs` не участвует
в автоматической LOCAL↔SERVER синхронизации (создаётся и живёт целиком на той БД, куда её накатили).
Для реальной работы фичи обязательна миграция на PROD.

### 1.2. Задать общий секрет и URL между karaoke-app и karaoke-web

Два internal-эндпоинта на karaoke-web (`/api/internal/stemjobs/{id}/raw` и `/ack`), по которым
karaoke-app забирает сырые файлы, защищены общим секретом в заголовке `X-Internal-Secret` — без
него любой, кто узнает адрес эндпоинта, мог бы читать чужие загруженные файлы. Нужно задать
**одинаковое** значение в двух разных местах:

**На стороне karaoke-web** (прод) — переменная окружения `STEMJOBS_INTERNAL_SECRET`:
1. Сгенерировать случайную строку, например: `openssl rand -hex 32`
2. Прописать её в `deploy/do.env` **на сервере** (этот файл не в git — так же, как
   `YOOKASSA_SHOP_ID`/`YOOKASSA_SECRET_KEY`, см. комментарии в
   `deploy/web-server-deploy/deploy/docker-compose-web.yml`):
   ```
   STEMJOBS_INTERNAL_SECRET=<сгенерированное_значение>
   ```
3. Передеплоить/перезапустить контейнер karaoke-web, чтобы переменная попала в окружение (обычный
   `do.sh`, деплой karaoke-web).

**На стороне karaoke-app** (эта машина) — через панель **«Настройки» (Properties) в webvue3**:
1. Открыть webvue3 → «Настройки» → найти ключ `stemJobsInternalSecret` → вставить **то же самое**
   значение, что в `STEMJOBS_INTERNAL_SECRET` на сервере → сохранить.
2. Там же найти ключ `stemJobsWebInternalUrl` → указать публичный базовый URL прод-сайта
   (например `https://sm-karaoke.ru`) — по нему karaoke-app обращается к karaoke-web за файлами.
   **Пока это поле пустое — поллер молча ничего не делает** (задания пользователей просто копятся
   в очереди на проде, без обработки и без ошибок в логах) — это осознанное поведение (тот же
   принцип, что у `sponsrSubscribersUrl`: пустая настройка = функция выключена, а не падает).

Пока секрет не задан **обеих сторонах одинаково**, `/api/internal/stemjobs/**` на karaoke-web
отвечает `403 Forbidden` (эндпоинты закрыты по умолчанию, а не открыты, если секрет не настроен).

### 1.3. Проверить требования к машине karaoke-app

Ничего нового не требуется — использует то же самое окружение, что и существующий демукс-пайплайн:
`docker` (образ `svoemestodev/demucs:latest` — уже используется для DEMUCS2/DEMUCS5), `ffmpeg` и
`ffprobe` в PATH (транскод входного файла в flac, перекодирование стемов в mp3, проверка
длительности). Если demucs для обычных песен у вас уже работает — для «Минусовок» дополнительно
ничего ставить не нужно.

## 2. Как проверить, что всё работает

1. Сделайте тестовому пользователю сайта премиум-доступ (панель «Пользователи сайта» в webvue3,
   `target=remote`).
2. Зайдите под этим пользователем в личный кабинет karaoke-public → «Создать минусовку» → загрузите
   короткий (1-2 минуты) аудиофайл, режим «Музыка + голос».
3. В течение ~45 секунд задание должно уйти из статуса «В очереди» (это видно и в личном кабинете
   пользователя, и в панели администратора «Минусовки» в webvue3, `target=remote`) — если этого не
   происходит, см. раздел «Типичные проблемы» ниже.
4. В панели «Процессы» webvue3 (`target=local`) должно появиться задание с именем `StemJob #<id>` —
   несколько быстро сменяющихся шагов (транскод, demucs, перекодирование в mp3), затем DONE.
5. В панели «Минусовки» статус должен смениться на «Готово» — скачайте пару стемов из личного
   кабинета пользователя, проверьте, что звук корректный.
6. Удалите тестовое задание (кнопка «Удалить» в любой из панелей) и убедитесь, что файлы пропали из
   MinIO (например, через консоль MinIO — `bucket "karaoke"`, префикс `stemjobs/<id>/`).

## 3. Панель администратора «Минусовки» (webvue3)

Пункт меню **«Минусовки»** — список всех заданий всех пользователей (переключатель БД сверху:
«Сервер» = реальные задания с прод-сайта, «Локальная» = для отладки на LOCAL). Колонки: пользователь,
файл, режим, статус, размер, ошибка (если есть), даты создания/истечения.

Действия у каждого задания:
- **«Остановить»** (доступно для «В очереди»/«В работе») — немедленно прерывает обработку: убивает
  docker-контейнер demucs (если он сейчас выполняется), удаляет оставшиеся шаги из очереди
  karaoke-app, помечает задание как «Ошибка» с текстом «Остановлено администратором». Файлы (если
  что-то уже успело загрузиться) не удаляются — для этого отдельная кнопка «Удалить».
- **«Удалить»** — то же самое, что делает автоматическая уборка по истечении 24 часов, но
  немедленно: если задание ещё выполняется — сначала останавливает (как выше), затем удаляет
  файлы из MinIO (если были) и саму запись из БД. Работает для задания в любом статусе.

То же самое доступно и пользователю из его личного кабинета (кнопка «Удалить» на своих заданиях) —
пользовательское удаление просто отмечает задание флагом, а фактическое удаление выполняет та же
периодическая уборка на karaoke-app (раз в 5 минут) либо администратор кнопкой «Удалить» немедленно.

Отдельной админ-панели для **правки** заданий (изменить режим/переставить в очереди и т.п.) нет —
это не предполагалось сценарием, только просмотр/остановка/удаление.

## 4. Мониторинг

- Светофор состояния проекта в шапке webvue3 покажет предупреждение **«Застрявшие задания
  „Минусовки“»**, если есть задания, которые дольше 30 минут находятся в «В очереди»/«В работе» без
  движения — обычно значит, что karaoke-app был выключен, либо поллер не настроен (см. 1.2), либо
  что-то зависло в docker.
- Обычная панель «Процессы» (`target=local`) показывает live-прогресс текущего выполняющегося
  задания «Минусовки» — тот же механизм и та же панель, что и для обычного демукса песен.

## 5. Типичные проблемы

| Симптом | Вероятная причина | Что делать |
|---|---|---|
| Задания вечно висят «В очереди», в логах karaoke-app тишина | `stemJobsWebInternalUrl` не задан (пусто) | Задать значение в Properties (см. 1.2) |
| В логах karaoke-web `403` на `/api/internal/stemjobs/**` | Секреты не совпадают либо не заданы | Сверить `STEMJOBS_INTERNAL_SECRET` (сервер) и `stemJobsInternalSecret` (Properties) — должны быть идентичны |
| Все задания сразу падают в «Ошибка» с текстом про длительность | На машине karaoke-app недоступен `ffprobe` (не в PATH) | Проверить, что `ffprobe` доступен из того же окружения, в котором запущен karaoke-app |
| Задание падает в «Ошибка»: «Обработка не создала все ожидаемые файлы» | Упал один из шагов demucs/ffmpeg (например, повреждённый/нераспознаваемый входной файл) | Смотреть лог задания в панели «Процессы» — там полный вывод команд |
| karaoke-app перезапущен во время обработки | Задание останется висеть «В работе» бесконечно (сама строка задания это не отследит) | Остановить его вручную кнопкой «Остановить» в панели «Минусовки» — дальше пользователь сможет создать новое |
| Файлы не удаляются по истечении 24 часов | karaoke-app был выключен всё это время — уборка не работает без него | Просто запустите karaoke-app — уборка досчитает пропущенное при следующем тике (раз в 5 минут) |

## 6. Известные ограничения этой версии (сознательно не делали)

- **Лимиты захардкожены в коде**, не редактируются через Properties: размер файла (50 МБ),
  длительность (60 минут), очередь на пользователя (5 заданий), срок хранения после готовности
  (24 часа) — константы `MAX_FILE_SIZE_BYTES`/`MAX_DURATION_SECONDS`/`MAX_ACTIVE_JOBS_PER_USER`/
  `RETENTION_HOURS` в `karaoke-app/.../model/StemJob.kt`. Чтобы поменять — правка кода и пересборка
  обоих модулей (karaoke-app + karaoke-web, лимиты дублируются на фронте karaoke-public тоже).
- Прямая загрузка файла из браузера в MinIO, минуя оба сервера (presigned PUT через доработку
  nginx-прокси на хосте) — не сделано, файл всегда проходит через диск karaoke-web.
- Остановленное администратором задание не уведомляет пользователя явно (он просто увидит статус
  «Ошибка» с текстом «Остановлено администратором» при следующем обновлении своей страницы).
