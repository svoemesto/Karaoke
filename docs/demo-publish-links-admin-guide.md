# Публикация DEMO-версии на VK/Dzen/Telegram/Max — инструкция для администратора

Ветка `feature/demo-publish-links` (запушена в origin, **пока не смержена в master** — ждём
подтверждения после вашего тестирования). Добавляет для DEMO-версии песни (короткий ознакомительный
фрагмент караоке, `RenderVersion.DEMO`) точно такой же функционал публикации на 4 площадках
(VK/Dzen/Telegram/Max), какой уже есть для LYRICS/KARAOKE/CHORDS/TABS в `SongEdit.vue`: ссылка,
версия, кнопки открыть/скопировать заголовок и текст поста/скопировать ссылку, undo/copy/paste,
автообрезка вставленной полной ссылки до голого id. Строка DEMO расположена в каждом из 4 блоков
площадок в порядке **KARAOKE → LYRICS → DEMO → CHORDS/TABS**, с отдельной иконкой (ножницы —
отличается от микрофона KARAOKE).

Заодно в этой ветке подвернулись два не связанных с DEMO, но важных исправления — см. пункты 4 и 5.

## 1. Получить изменения

```bash
cd ~/Karaoke
git fetch origin
git checkout feature/demo-publish-links
git pull
```

Если у вас на машине есть незакоммиченные правки — сохраните их (`git stash`) перед переключением
ветки.

## 2. Применить миграцию БД

Файл `deploy/karaoke-db/23_demo_publish_links.sql` — новые колонки (`id_dzen_demo`/`version_dzen_demo`
и аналоги для vk/telegram/max) в `tbl_settings` **и** `tbl_settings_sync`, плюс индексы. Применить
**вручную на каждой БД отдельно** (миграция сама на сервер не попадает — как и остальные файлы в
`deploy/karaoke-db/`):

```bash
# LOCAL
psql -h localhost -p 8832 -U postgres -d karaoke -f deploy/karaoke-db/23_demo_publish_links.sql

# PROD (когда будете готовы выкатывать на прод — после подтверждения тестирования)
psql -h <PROD_SERVER_IP> -p 8832 -U postgres -d karaoke -f deploy/karaoke-db/23_demo_publish_links.sql
```

Затем **обязательно** переприменить (эти два файла — источник правды для триггеров recordhash,
пересоздают функции через `CREATE OR REPLACE`, идемпотентны, безопасно гонять повторно) — новые
колонки участвуют в синхронизации LOCAL↔SERVER, без этого шага изменения в полях DEMO не будут
попадать в hash сравнения при синхронизации:

```bash
psql -h localhost -p 8832 -U postgres -d karaoke -f deploy/recordhash_settings.sql
psql -h localhost -p 8832 -U postgres -d karaoke -f deploy/recordhash_settings_sync.sql
```

(на PROD — аналогично, с `-h <PROD_SERVER_IP>`)

## 3. Пересобрать и перезапустить karaoke-app и webvue3

**Важно:** не используйте `build_start_app`/`build_start_webvue3`/`do.sh start`/`do.sh load` без
параметра — это generic-команды, которые трогают/пересоздают **весь** стек контейнеров разом через
устаревший `docker-compose.yml` (в котором, в частности, для `karaoke-web` не хватает части
переменных окружения — контейнер падает в CrashLoop). Пользуйтесь **точечными** командами:

```bash
cd ~/Karaoke/deploy

./do.sh build_app
./do.sh start_app

./do.sh build_webvue3
./do.sh start_webvue3
```

### Если на вашей машине нет NVIDIA GPU (passthrough)

`karaoke-app` в `docker-compose-app.yml` резервирует NVIDIA GPU для аппаратного кодирования видео.
Если у вас его нет (например, тестируете в VM без GPU-passthrough) — `start_app` упадёт с ошибкой
`could not select device driver "nvidia"`. Решение — новый переключатель `ENABLE_APP_GPU` в
`deploy/do.env` (файл в `.gitignore`, у каждой машины свой):

```
ENABLE_APP_GPU=0
```

По умолчанию (если строки нет вообще, или на машинах с реальным GPU) — `1`, поведение как раньше,
ничего менять не нужно. `0` запускает `karaoke-app` без GPU-реквизита (`deploy/docker-compose-app.gpu.yml`
просто не подключается) — рендеринг видео с аппаратным ускорением в этом режиме не тестировался, но
для проверки самой фичи публикации DEMO (см. ниже) GPU не требуется вообще.

## 4. Побочный фикс: nginx в webvue3 (важно, если раньше видели непонятные 404 в админке)

`webvue3/nginx_webvue3.conf`, блок `location /api` — `proxy_pass` с переменной обрезал **любой**
запрос `/api/*` до буквального `/api`, вообще без остального пути. На практике это ломало **всю**
админку целиком (список песен не грузился, SSE-подписка на события 404-ила в консоли и т.п.), а не
только DEMO. Исправлено (`proxy_pass $karaoke_app_upstream$request_uri`). Если раньше у вас в консоли
браузера в webvue3 были повторяющиеся 404 на `/api/subscribe`, `/api/songsdigests` и т.п. — после
обновления они должны исчезнуть. Отдельных действий не требует, применяется автоматически при
пересборке `webvue3` (шаг 3).

## 5. Побочный фикс: `deploy/backup_db.sh`

Мелкий баг: `"~/Karaoke/deploy/karaoke-db/..."` в закавыченном пути — `~` в bash не раскрывается
внутри двойных кавычек, backup мог падать/писать не туда. Заменено на `$HOME`. Если пользуетесь этим
скриптом на сервере — не забудьте засинхронизировать его туда (см. `DEVELOPMENT.md`, раздел
«Синхронизация серверных конфигов»); я не смог сделать это сам — нет настроенного SSH-доступа к
проду в этой среде.

## 6. Как тестировать

1. Откройте webvue3 → «Песни», кликните по любой песне (открывается `SongEdit`).
2. Для **каждой** из 4 площадок (иконки Dzen/VK/Telegram/Max) проверьте, что строка **DEMO**:
   - находится в порядке **KARAOKE → LYRICS → DEMO → CHORDS/TABS** (сразу после LYRICS, перед CHORDS);
   - иконка — «ножницы» (не микрофон KARAOKE и не нота LYRICS);
   - видна **всегда**, без скрытия (в отличие от CHORDS/TABS, которые прячутся, если пусты).
3. Вставьте в поле ссылки DEMO полную ссылку с площадки (например, `https://vkvideo.ru/video12345`)
   — после потери фокуса поле должно автоматически обрезаться до голого id (`12345`), а поле версии —
   проставиться текущим значением «Версия» песни.
4. Кнопка **Head** (для всех 4 площадок) и **Body** (только Dzen/VK) — должны скопировать в буфер
   текст с меткой `Demo`/`Karaoke (Demo)` (без ошибок в консоли, без 404 в Network).
5. Кнопка **Undo** — должна возвращать поле к последнему сохранённому значению.
6. Сохраните (дождитесь автосохранения, по умолчанию ~1 сек после последнего изменения) и
   перезагрузите страницу — введённые id/версия DEMO должны сохраниться.
7. Проверьте, что ничего не сломалось у соседних версий (LYRICS/KARAOKE/CHORDS/TABS) — тот же набор
   кнопок, то же поведение, что и раньше.
