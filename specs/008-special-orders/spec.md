# Feature Specification: «Отдельные песни разных авторов» — виртуальная плашка в Закромах

**Feature Branch**: `[008-special-orders]`
**Created**: 2026-07-24
**Status**: Draft (концепция **пересмотрена** после обсуждения)
**Input**: User description: "перенос одиночных песен в отдельный раздел — последняя
'плашка' в закромах (как будто это автор), структура внутри: Автор → Альбом → Песни"

## ⚠️ Изменение концепции

**Первоначально планировалась** отдельная страница `/special-orders`. После обсуждения
**решено иначе**:
- ❌ **НЕ** создаём `/special-orders` (отдельная страница)
- ❌ **НЕ** создаём `SpecialOrdersView.vue`
- ✅ **ДА**: виртуальная плашка внутри `ZakromaView.vue`, как **последний «автор»** в списке
- Структура внутри плашки: **Автор → Альбом → Песни** (та же иерархия, что и у реальных авторов)

## Стратегический контекст

**Гипотеза H1.20** (см. [`docs/strategy/growth-audit.md`](../growth-audit.md)):
есть несколько авторов с 1-2 песнями (по спецзаказу) — для них нужно убрать из основного
списка «Закромов» и перенести в **виртуальную плашку**, чтобы пользователь видел
«отдельные песни разных авторов» как **самостоятельную категорию**, а не вперемешку
с авторами с полной дискографией.

**M-23** в roadmap.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Зарег/аноним видит «виртуального автора» в Закромах (Priority: P1)

Зарегистрированный или анонимный пользователь на странице `/zakroma` после списка
**всех реальных авторов** (с полной дискографией) видит **последнюю** «плашку» с
названием «Отдельные песни разных авторов». Внутри — та же структура, что и у обычного
автора: под-авторы → альбомы → песни.

**Why this priority**: основная фича. Без неё весь смысл пропадает.

**Independent Test**: открыть `/zakroma`, проскроллить до конца, увидеть последнюю плашку
с «Отдельные песни разных авторов».

**Acceptance Scenarios**:
1. **Given** пользователь на `/zakroma`, **When** листает до конца, **Then** видит
   последнюю плашку с подзаголовком «Отдельные песни разных авторов».
2. **Given** пользователь на этой плашке, **When** разворачивает любого «под-автора»,
   **Then** видит список его песен (1-2 шт) с теми же колонками, что и для обычного автора.
3. **Given** пользователь на плашке, **When** кликает на песню, **Then** переходит на
   `/song?id=N` (как обычно).

---

### User Story 2 - Спецзаказные авторы скрыты из основного списка (Priority: P1)

В `ZakromaView.vue` в основном списке авторов (по алфавиту) **нет** спецзаказных.
Они есть **только** в виртуальной плашке «Отдельные песни разных авторов».

**Why this priority**: иначе каталог «Закрома» остаётся запутанным (mix полных и одиночных).

**Independent Test**: проверить, что среди V (ВИНО) и Y (ЯСВЕНА) в Закромах нет
спецзаказных авторов «Princesse Angine», «Инструкция по выживанию» и т.п.

**Acceptance Scenarios**:
1. **Given** пользователь на `/zakroma`, **When** смотрит список авторов, **Then** НЕ видит
   авторов с `tbl_authors.is_special_order=true`.
2. **Given** аноним/зарег., **When** ищет спецзаказного автора по имени через `/filter`,
   **Then** поиск находит его (отдельный flow), но в `/zakroma` его нет.

---

### User Story 3 - На странице автора видна пометка «По спецзаказу» (Priority: P2)

`AuthorPlaylistView.vue` (страница конкретного автора через `/author-playlist?name=X`)
для спецзаказного автора показывает пометку «По спецзаказу» вверху страницы.

**Why this priority**: ясность — пользователь, попавший на страницу автора через поиск
или прямой URL, видит контекст.

**Independent Test**: открыть `/author-playlist?name=Princesse Angine`, увидеть пометку
«По спецзаказу».

**Acceptance Scenarios**:
1. **Given** пользователь на `/author-playlist?name=X` (где X — спецзаказной автор),
   **When** страница загружается, **Then** видит пометку «По спецказу» вверху страницы.

---

### User Story 4 - Метрика: трекинг просмотров (Priority: P2)

Каждый заход на `/zakroma` пишется в `tbl_events` через `useEngagementTracking('zakroma')`
(уже есть). Клик на песню в виртуальной плашке пишется через стандартный трекинг.

**Independent Test**: проверить в `tbl_events`, что клики на песни из плашки фиксируются.

**Acceptance Scenarios**:
1. **Given** пользователь на `/zakroma`, **When** кликает на песню из виртуальной плашки,
   **Then** в `tbl_events` пишется `linkName` с `songName`.

## Requirements *(mandatory)*

### Functional Requirements

**Бэкенд:**

- **FR-001**: Добавить в `tbl_authors` колонку `is_special_order BOOLEAN DEFAULT FALSE NOT NULL`.
  Колонка входит в `recordhash` (как в `12_site_user_limits.sql`).
- **FR-002**: SQL миграция (одна, на LOCAL и PROD отдельно):
  ```sql
  ALTER TABLE public.tbl_authors
    ADD COLUMN IF NOT EXISTS is_special_order BOOLEAN DEFAULT FALSE NOT NULL;

  CREATE OR REPLACE FUNCTION public.update_tbl_authors_recordhash() RETURNS trigger
      LANGUAGE plpgsql
  AS $$
  BEGIN
      NEW.recordhash = md5(
          COALESCE(NEW.id::TEXT, '') ||
          COALESCE(NEW.author, '') ||
          ... (все имеющиеся поля) ... ||
          COALESCE(NEW.is_special_order::TEXT, '')
      );
      RETURN NEW;
  END;
  $$;

  UPDATE public.tbl_authors SET recordhash = md5(...) WHERE id > 0;
  ```
- **FR-003**: Backfill (в той же миграции): пометить `is_special_order=true` для авторов,
  у которых `count(*) < 3` песен в коллекции:
  ```sql
  UPDATE public.tbl_authors SET is_special_order = true
  WHERE id IN (
      SELECT author_id FROM public.tbl_settings
      WHERE id_status >= 3 AND btrim(author) != ''
      GROUP BY author_id
      HAVING count(*) < 3
  );
  ```
  (если `tbl_settings` не хранит `author_id`, то по `author` — уточнить схему).
- **FR-004**: В `PublicApiController` метод `GET /api/public/authors` поддерживает
  параметр `?scope=main|special|all` (default=`main`):
  - `main` — авторы БЕЗ `is_special_order=true`
  - `special` — авторы С `is_special_order=true`
  - `all` — все (для админки)
- **FR-005**: В `PublicApiController` метод `GET /api/public/authors-tiles` поддерживает
  тот же `?scope=`.
- **FR-006**: В `tbl_authors` API возвращает поле `isSpecialOrder` в `AuthorTilePublicDto`.
- **FR-007**: API `/api/public/zakroma?specialBucket=true` (новый параметр существующего
  endpoint) возвращает песни всех спецзаказных авторов одним запросом (без N+1 по авторам),
  в структуре ZakromaPublicDto (автор → альбом → песни). Реализовано через
  `Zakroma.getZakromaBySpecialOrder()` — см.
  [docs/features/special-orders.md](../../docs/features/special-orders.md).

**Фронт:**

- **FR-101**: В `ZakromaView.vue` — после основного списка авторов (в `zak`) рендерится
  **виртуальная плашка** «Отдельные песни разных авторов». Внутри — данные из нового endpoint.
- **FR-102**: Виртуальная плашка имеет:
  - Заголовок: «Отдельные песни разных авторов»
  - Под-авторы (реальные спецзаказные) с альбомами и песнями
  - Структура таблиц — как у обычных авторов (ZakromaView.vue:121-200)
- **FR-103**: `ZakromaView.vue` — фильтр (FR-004): в основной список загружаются только `scope=main`.
- **FR-104**: `AuthorPlaylistView.vue` — пометка «По спецзаказу» вверху, если автор спецзаказной.
- **FR-105**: JSDoc на новых функциях (FR-006) с `@see` ссылками на
  `docs/strategy/growth.md`, `growth-audit.md`.

### Non-Functional Requirements

- **NFR-001**: Адаптивный дизайн (mobile + desktop), как `ZakromaView`.
- **NFR-002**: Семантический HTML.
- **NFR-003**: ESLint clean, JSDoc coverage 100%.
- **NFR-004**: Pre-commit 7 проверок зелёные, CI 7/7 PASS.
- **NFR-005**: Никаких изменений в `tbl_settings` (только `tbl_authors`).

### Key Entities

- **`tbl_authors.is_special_order`** (новое) — boolean, default false
- **`AuthorTilePublicDto.isSpecialOrder`** (новое поле DTO) — boolean
- **Виртуальная «плашка» (frontend)** — ZakromaSpecialBucket (только в коде, нет в БД)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001 (главная)**: виртуальная плашка «Отдельные песни разных авторов» видна в `/zakroma`
  последней в списке, содержит всех спецзаказных авторов с их песнями.
- **SC-002**: 0 спецзаказных авторов в основном списке `/zakroma` (только в виртуальной плашке).
- **SC-003**: ≥10% посетителей `/zakroma` дочитывают до виртуальной плашки (scroll-веха 75%+).
- **SC-004**: ≥5% посетителей `/zakroma` кликают на песню из виртуальной плашки.
- **SC-005**: среднее время на `/zakroma` не падает после фичи (не замедляет каталог).
- **SC-006**: страница адаптивна, загружается <2 сек.

## Assumptions

- **Backfill**: `is_special_order=true` для авторов с `count(*) < 3` песен — определяется
  по `tbl_settings WHERE id_status>=3 GROUP BY author HAVING count(*) < 3`.
- **Дизайн**: виртуальная плашка стилизована как обычный «автор», но с заметным отличием
  (например, иконка группы/папки, другой фон).
- **Админка**: ручное управление `is_special_order` — отдельная фича (webvue3).
- **Аналитика**: тот же трекинг `/zakroma` (отдельный endpoint не нужен).

## Out of Scope (явно не в этой фиче)

- ❌ **Отдельная страница `/special-orders`** — концепция отменена, НЕ делаем
- Подписка на новые спецзаказы (отдельная фича)
- Возможность «заказать свою песню» (нет услуги)
- Подсчёт «сколько заказов у автора» (не нужно)
- Ручное управление `is_special_order` в webvue3 (отдельная фича)
- Email-уведомления о новых спецзаказах (отдельная фича)

## Открытые вопросы (закрыты по факту реализации)

1. **Иконка**: 📁 (закреплено в коде, `.km-special-tile-icon`).
2. **Обложка**: плейсхолдер-иконка вместо фото (пусто в данных — не блокер).
3. **Подзаголовок**: убран по фидбэку пользователя — плашка обходится без поясняющего

5. **`/author-playlist` для спецзаказного**: только баннер «По спецзаказу» вверху, остальное
   как у обычного автора — реализовано (User Story 3).
