# Feature Specification: Таблица «FREE vs PREMIUM» на /premium

**Feature Branch**: `[005-free-vs-premium]`
**Created**: 2026-07-24
**Status**: Draft
**Input**: User description: "Free-пользователь не понимает, что получит за деньги. На /premium
должна быть таблица 'FREE vs PREMIUM' с фичами."

## Стратегический контекст

**Гипотеза** (см. [`docs/strategy/growth.md`](../growth.md)): free-зарегистрированный пользователь
не понимает, какие преимущества даёт премиум → не покупает. Конверсия registration→premium
сейчас 15% (3 из 20), и из них многие покупают «вслепую».

**QW-1** — третья фича в Top-3. Гипотеза H2.1 из аудита.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Free-пользователь видит сравнение FREE vs PREMIUM (Priority: P1)

Free-зарегистрированный пользователь (или аноним) переходит на `/premium` и видит
**таблицу сравнения** «FREE vs PREMIUM» с понятными фичами:

| Фича | FREE | PREMIUM |
|------|------|---------|
| Просмотр эфирных песен (видео VK) | ✅ | ✅ |
| Поиск и каталог | ✅ | ✅ |
| Демо-фрагменты эксклюзивных | ✅ | ✅ |
| Полный доступ к плееру (все песни) | ❌ | ✅ |
| Избранное | до 100 | без лимита |
| Свои плейлисты | 1 (избранное) | до 50 |
| Скачивание MP4 | ❌ | ✅ (3 версии) |
| Создание минусовок | ❌ | ✅ |
| Чат с автором проекта | ❌ | ✅ |
| Без рекламы / скидки на премиум | — | ✅ |

**Why this priority**: без явного сравнения пользователь не понимает ценность → не покупает.
Корень проблемы registration→premium.

**Independent Test**: free-пользователь заходит на `/premium`, видит таблицу, в 1 из 5 (20%)
кликает на «Оформить» (можно отследить через `tbl_events`).

**Acceptance Scenarios**:
1. **Given** аноним или free-пользователь, **When** открывает `/premium`, **Then** под заголовком
   видит таблицу сравнения «FREE vs PREMIUM» с ≥10 строками.
2. **Given** free-пользователь на `/premium`, **When** сравнивает FREE и PREMIUM, **Then** понимает
   разницу (зелёные галочки / красные кресты, выделение PREMIUM-колонки).
3. **Given** free-пользователь на `/premium`, **When** решает оформить, **Then** внизу таблицы — CTA
   «Оформить премиум-подписку» (ведёт на блок выбора тарифа, который уже есть).

---

### User Story 2 - Премиум-пользователь видит «Спасибо» (Priority: P3)

Премиум-пользователь, возвращающийся на `/premium`, видит упрощённый вариант:
таблица показывается с пометкой «Вы премиум-пользователь — спасибо!» + блок
«Управление подпиской».

**Why this priority**: для премиум блок «сравнения» уже не мотивирует — они и так премиум.
Но прятать нельзя — там же управление подпиской. P3 — потому что не блокер.

**Independent Test**: премиум-пользователь заходит на `/premium`, видит таблицу с пометкой
«Вы уже премиум», внизу — ссылки на «Мои подписки», «Избранное», «Плейлисты».

**Acceptance Scenarios**:
1. **Given** премиум-пользователь, **When** открывает `/premium`, **Then** видит таблицу
   сравнения с пометкой «Вы премиум» + блок «Что вы получили» (с акцентом на используемые фичи).
2. **Given** премиум-пользователь, **When** нажимает «Управление подпиской», **Then** переходит
   на `/account/subscriptions` (уже есть).

---

### User Story 3 - Метрика: трекинг конверсии (Priority: P2)

Каждый клик на «Оформить» из таблицы пишется в `tbl_events` (event_type='ui',
link_type='navigate', link_name='free_vs_premium_cta'). Чтобы измерить конверсию таблицы.

**Why this priority**: без метрики не узнаем, работает ли таблица. P2 — это инфра
измерения, не сам блок.

**Independent Test**: проверить в БД, что событие пишется при клике.

**Acceptance Scenarios**:
1. **Given** free-пользователь, **When** кликает «Оформить» в таблице, **Then** в `tbl_events`
   появляется запись (event_type='ui', link_type='navigate', link_name='free_vs_premium_cta').

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: На странице `/premium` (`PremiumView.vue`) добавить таблицу сравнения «FREE vs PREMIUM».
- **FR-002**: Таблица содержит **ровно 2 колонки** (FREE, PREMIUM) и **≥10 строк-фичей**.
- **FR-003**: Список фичей (предлагаемый, финальный — с пользователем):

  | # | Фича | FREE | PREMIUM |
  |---|------|------|---------|
  | 1 | Просмотр эфирных песен (видео VK) | ✅ | ✅ |
  | 2 | Поиск и каталог | ✅ | ✅ |
  | 3 | Демо-фрагменты эксклюзивных | ✅ | ✅ |
  | 4 | **Полный доступ к плееру** (все песни) | ❌ | ✅ |
  | 5 | **Избранное** | до 100 песен | без лимита |
  | 6 | **Свои плейлисты** | 1 (избранное) | до 50 |
  | 7 | **Скачивание MP4** (3 версии) | ❌ | ✅ |
  | 8 | **Создание минусовок** (Demucs) | ❌ | ✅ |
  | 9 | **Чат с автором проекта** | ❌ | ✅ |
  | 10 | **История прослушиваний** | ❌ | ✅ |
  | 11 | **Уведомления о новых песнях** | ❌ | ✅ |
  | 12 | **Без рекламной метки** при шаринге | ❌ | ✅ |

- **FR-004**: Визуально выделить PREMIUM-колонку (фон, рамка, иконка короны).
- **FR-005**: Внизу таблицы — CTA «Оформить премиум-подписку» (или «Перейти к тарифам»).
- **FR-006**: Для премиум-пользователей — пометка «Вы премиум» + блок «Что вы получили».
- **FR-007**: Адаптивный дизайн: десктоп — 3 колонки (фича / FREE / PREMIUM), мобильный — стопка
  (фича → «FREE: да/нет» → «PREMIUM: да/нет»).
- **FR-008**: Клик на CTA пишется в `tbl_events` (event_type='ui', link_type='navigate',
  link_name='free_vs_premium_cta').
- **FR-009**: Использует существующий дизайн (`--km-*` CSS-переменные).
- **FR-010**: Dual classic/modern (если применяется к странице).
- **FR-011**: Список фичей хранится в исходниках (`.vue`), не в БД (для простоты).

### Non-Functional Requirements

- **NFR-001**: Таблица не должна замедлять загрузку `/premium` (inline, без запросов).
- **NFR-002**: Семантический HTML (`<table>`, `<thead>`, `<tbody>`, `<th>`, `<td>`).
- **NFR-003**: Контрастность WCAG AA.

### Key Entities

Нет новых сущностей. Переиспользуем:
- `tbl_events` — для трекинга (уже есть).
- `useAuth().user.effectivePremium` — для определения премиум (уже есть).
- `PremiumView.vue` — точка размещения.
- `tbl_price_tariffs` — для отображения тарифов (уже есть).
- `tbl_subscriptions` — для проверки статуса подписки (уже есть).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001 (главная)**: конверсия registration→premium **с 15% до 20%** за 1 месяц после запуска
  (рост ×1.33). Это +1 премиум/мес.
- **SC-002**: ≥30% free-пользователей, зашедших на `/premium`, **дочитывают до таблицы**
  (scroll-веха 75%).
- **SC-003**: ≥10% free-пользователей, увидевших таблицу, **кликают на CTA** (через `tbl_events`).
- **SC-004**: среднее время на `/premium` не падает (не раздражает).
- **SC-005**: таблица адаптивна (тестируется на 360px, 768px, 1280px).

## Assumptions

- Список фичей (12 штук) — пишется совместно (агент предлагает, пользователь утверждает).
- Дизайн dual classic/modern — переиспользуем существующие компоненты.
- Премиум-пользователь определяется через `useAuth().user.effectivePremium` (уже есть).
- Трекинг через `trackUi('navigate', 'free_vs_premium_cta')` (уже есть).
- Не требует изменений в БД.
- Блок не зависит от других QW-фич (QW-9, QW-2) — может идти параллельно.
- Существующий блок выбора тарифа (в `PremiumView.vue:21-103`) остаётся — таблица
  добавляется **над** ним.

## Out of Scope (явно не в этой фиче)

- A/B-тест формулировок фичей (отдельная фича, L-1).
- Динамический список фичей из БД (всё в коде).
- Админ-редактор (в коде).
- Видео-презентация премиума.
- Тест-драйв на 1 день (отдельная фича, отложено).
- «Семейный» тариф (отдельная фича, M-21).

## Открытые вопросы

1. **Точный список фичей** — обсудить с пользователем. 12 предложенных — что убрать/добавить?
2. **Дизайн таблицы** — обычная `<table>`, карточки-сравнения, или 2-колоночный grid? (рекомендую `<table>` для первого релиза — проще и адаптивнее).
3. **Где разместить таблицу** — над тарифами, под ними, между? (рекомендую **над** тарифами, чтобы сначала «зацепить» фичами, потом «продать» цену).
4. **Какая версия таблицы для премиум** — «Вы получили всё это» или полная таблица с галочками? (рекомендую «Вы получили всё это» + выделить строки с фичами, которые премиум реально использует).
5. **Анимация** — fade-in для строк таблицы при скролле? (рекомендую без анимации для первого релиза).
6. **Привязка к тарифам** — показывать рядом с тарифом «включает: 12 фичей выше», или нет? (рекомендую нет — тарифы отдельный блок).

- **M-1** (Воронка в webvue3) — нужен для измерения SC-001.
- **M-2** (event_type='subscription_paid') — нужен для измерения SC-001.
